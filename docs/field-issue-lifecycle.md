# Field Issue Lifecycle

## Practical Sequence

1. Issue is captured by a superintendent, technician, PM, or support lead.
2. Ownership is assigned based on who can move the work next.
3. A due date is set to avoid open-ended follow-up.
4. Severe or aging items are escalated.
5. Once resolved, the issue is closed or linked to a closeout requirement if turnover work remains.

## Working Assumptions

- issues should stay specific and action-oriented
- not every issue is severe, but every issue should have a visible owner
- punch and closeout items should connect back to the same project-level operating picture
